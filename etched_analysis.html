Etched Sohu: Continuous Batching Throughput Analysis

Reconstructing and stress-testing Etched's thesis that LLM inference is compute-bound under continuous batching. All formulas derived from first principles. Model: Llama 3 70B (68.45B actual parameters).
By Abhishek Lal  |  Source: Etched — "Making the Biggest Bet in AI" (archived; video)  |  Sohu: 8-chip system, 144 GB HBM3 per chip
Etched's Thesis

Etched claims "Transformers are compute-bound" under continuous batching.

The core idea of Etched revolves around transformers being a compute-bound workload. Etched states that since it is compute-bound, adding FLOPS helps with throughput. The key question is how much performance can be gained. Etched states that if hypothetically you have infinite compute on chip, then the bottleneck would again be shifted back to HBM and total token/sec would be governed by HBM bandwidth. For every batch processing, it would fetch weights and KV cache data of all decode users from HBM. The number of times it can repeat this cycle times batch size determines the overall token/sec of the 8-chip system. This calculation would give an upper bound on the throughput of such a system.

Assumptions:
(a) Infinite on-chip compute with no internal bottlenecks — the only constraint is HBM bandwidth.
(b) The model is perfectly parallelized across 8 chips, each containing HBM3 stacks.

Under these assumptions, Etched selects ISL=2048, OSL=128 as the benchmark workload, which they describe as "NVIDIA and AMD's standard benchmark". With batch size 128, continuous batching produces 1 prefill user occupying 2,048 slots in the matrix computation and 127 decode users each occupying 1 slot — totaling 2,175 token slots per iteration.

All linear matrix operations share the model weights across the batch. However, each of the 127 decode users must also fetch their individual KV caches from HBM, which cannot be shared.

Total HBM reads per iteration = model weights + 127 users' KV caches. Since compute is assumed infinite, the time to process 2,048 input tokens + 127 output tokens is governed entirely by the time to complete these HBM reads.

The question this dashboard answers: does this arithmetic intensity actually exceed the machine's compute-to-bandwidth ratio — and does it hold when you vary the ISL/OSL split?

04 — Are Real Workloads Like Etched's Benchmark?

TL;DR

Etched claims LLM inference is compute-bound — that you need massive TFLOPS. They benchmark at ISL=2048, OSL=128, cache=0, FP8 dense. Real-world production data (OpenRouter, 100T+ tokens) shows an aggregate 80:1 input-to-output ratio. But ~80% of those input tokens are cached KV (system prompts, RAG context, conversation history) — decomposing the 80:1 into ~16 fresh ISL : ~64 cached prefix : 1 OSL. Etched models only the 16:1 fresh ratio and ignores the 64 parts of cached KV that are read from HBM every decode iteration. Adding a realistic cache drops Sohu throughput significantly. Moving to real-world workloads (coding agents, reasoning, agentic) drops it further — see the hardware comparison table. Blackwell (B200) like config at FP4+sparse with 8.0 TB/s absorbs cache pressure better — but even there, the coding workload cuts throughput significantly vs the synthetic benchmark. The hardware is good. The thesis is wrong. Sohu is fast because it has bandwidth, not because inference is compute-bound. See the hardware comparison table below.

Why Cached Prefix KV (Cache Reads) Matters So Much

  • Every decode user, every iteration reads the full cached prefix KV (cache reads) from HBM. With U=127 decode users and a 6,912-token cache, that's 127 × 6,912 × 165 KB/tok ≈ 148 GB of extra HBM reads per iteration at FP8 KV.
  • This is additive and constant. Unlike the triangular decode KV (which grows with OSL), the cached prefix KV (cache reads) cost is the same whether OSL=128 or OSL=8192. It's a flat tax on every iteration.
  • Prompt caching saves billing, not bandwidth. Anthropic charges 90% less for cached reads. But the KV data still sits in HBM and is read at full bandwidth cost. The economic optimization and the hardware optimization are completely different.
  • The workload is more memory-bound than Etched claims. Adding cache pressure increases HBM reads without proportionally increasing FLOPs, which drops arithmetic intensity. This means the workload needs more bandwidth, not more TFLOPS — undermining Etched's core argument that inference is compute-bound.
  • Weight compression is the real unlock. At FP4+sparse, weights shrink from 70.6→26.7 GB — freeing bandwidth for KV reads. This is why Blackwell with FP4+sparse maintains high throughput even with cache pressure. But as the coding workload shows, even that headroom gets consumed by longer context. See the hardware comparison table for the full picture.

The paradox

  • Etched claims LLM inference is compute-bound. Their thesis: arithmetic intensity exceeds GPU balance points → you need massive TFLOPS → buy our ASIC. At their benchmark config (ISL=2048, OSL=128, cache=0, FP8 dense), arithmetic intensity is high — above typical GPU balance points.
  • But cache pressure pushes workloads in the opposite direction. Adding a realistic 6,912-token cached prefix (cache reads) dramatically increases HBM reads (denominator) without proportionally increasing FLOPs (numerator). Arithmetic intensity drops. The workload becomes more memory-bound, not less.
  • At FP8 dense + cache, their own hardware suffers. Same ISL:OSL, same 8 Sohu chips — adding cache dramatically increases HBM reads. The cache penalty scales with decode users × cache tokens × λ.
  • But adopt Blackwell-era formats (FP4 + 2:4 sparsity) and the picture flips. Weights shrink from 70.6→26.7 GB. On B200 (8.0 TB/s per chip), even with cache pressure, the system maintains high throughput. The hardware is excellent — but the throughput comes from bandwidth freed by weight compression, not from massive TFLOPS. The compute-bound thesis is irrelevant to why it's fast.
  • Move to real workloads and it drops further. Coding agents, reasoning models, and agentic workflows all have large cache prefixes and longer outputs — every platform sees significant throughput drops vs the synthetic benchmark. See the hardware comparison table for exact numbers.
  • This reveals the real bottleneck. At FP4+sparse, model weights are ~27 GB vs hundreds of GB of KV cache reads. The system is overwhelmingly KV-bandwidth-bound. More TFLOPS doesn't help — you need more HBM bandwidth and smarter KV management (quantized KV, paged attention, speculative decoding).
  • Etched's ISL:OSL ratio matches real traffic. The aggregate ~80:1 (or 16:1 fresh ISL:OSL) is consistent with OpenRouter data. This is not where their analysis goes wrong.
  • The fastest-growing workloads (reasoning, coding, agents) are output-heavy — ISL:OSL of 1:1 to 1:8 — where decode dominates and per-user throughput matters more than aggregate tok/s. These workloads have even lower arithmetic intensity.
  • Summary: Sohu's 3.35 TB/s HBM bandwidth per chip is genuinely valuable. But it's valuable because inference is memory-bound — the exact opposite of what Etched claims. The TFLOPS they advertise is solving the wrong bottleneck.

05 — Derivation

Llama 3 70B Architecture

  • Dimensions: d_model=8192, n_layers=80, n_heads=64, n_kv_heads=8 (GQA), head_dim=128, ffn_dim=28672 (SwiGLU)
  • Per-layer params: Attention = 150,994,944, FFN = 704,643,072
  • Total params: 80 × (150.99M + 704.64M) = 68,451,041,280 ≈ 68.45B

KV Cache Per Token (λ)

  • Base elements: n_layers × n_kv_heads × head_dim × 2 = 80 × 8 × 128 × 2 = 163,840 elements per token (×2 for K and V)
  • Block scaling: FP4/FP8 use microscaling — 1 byte E8M0 exponent shared per 32 elements → effective bytes/val = precision + 1/32
  • At FP4 + block scale: 163,840 × (0.5 + 1/32) = 163,840 × 0.53125 = 87,040 bytes (85 KB/tok)
  • At FP8 + block scale: 163,840 × (1.0 + 1/32) = 163,840 × 1.03125 = 168,960 bytes (165 KB/tok)
  • At FP8: 163,840 × 1.03125 = 168,960 bytes (165 KB/tok) — Etched (FP8 + block scaling)
  • At FP16: 163,840 × 2.0 = 327,680 bytes (320 KB/tok) (no block scaling needed; full per-element exponent)

Model Weights in HBM (M)

  • Dense formula: M = total_params × (precision + block_scale_overhead)
  • Sparse formula: M = total_params × 0.5 × (precision + 1/32) + total_params × 0.125 — half the values stored + sparsity metadata (4 bits per 4 dense elements, i.e. 2×2-bit indices)
  • FP4 + 2:4 sparse: 68.45B × 0.5 × 0.53125 + 68.45B × 0.125 = 18.18 + 8.56 = 26.74 GB
  • FP4 dense: 68.45B × 0.53125 = 36.36 GB
  • FP8 dense: 68.45B × 1.03125 = 70.59 GB — Etched (FP8 + block scaling)
  • FP16 dense: 68.45B × 2.0 = 136.90 GB (no block scaling; full per-element exponent)

Steady-State Continuous Batching

  • Sequence length: sl = ISL + OSL — ISL is fixed (prompt length), OSL varies (output length), total grows
  • Decode users: U = ⌊OSL / sl × BS⌋ — fraction of batch in decode phase
  • Staggered KV cache: user i has context (cacheTokens + ISL + i), so KV[i] = (cacheTokens + ISL + i) × λ for i = 0, 1, ..., U−1
  • Effective context: cacheTokens + ISL + decode_pos — each decode user attends to the full cached prefix (cache reads) + fresh ISL + their accumulated output tokens

HBM Data Per Iteration

  • Reads: M + (C_kv + KV[0])×U + λ×U(U−1)/2 — weights + cached prefix KV (cache reads) + ISL KV + triangular decode KV per user
  • Writes: KV[0] + λ×U — prefill user's new KV + decode users' new tokens (cached prefix KV (cache reads) already stored)
  • Total: HBM = M + C_kv×U + KV[0]×(U+1) + λ×U×(U+1)/2
  • C_kv = cacheTokens × λ — KV bytes for cached prefix (cache reads: system prompt, RAG context). Pre-computed but read from HBM every iteration for every decode user.
  • Note: Prompt caching saves prefill compute (no re-encoding) but NOT decode bandwidth. Every user still reads the full cached KV from HBM each step.

Throughput (Memory-Bound)

  • Time per iteration: T = HBM / BW
  • Output tok/s: BW × U / HBM
  • Input tok/s: BW × (BS − U) / HBM
  • Total tok/s: BW × BS / HBM
  • Req/s: Out_tok/s / OSL
  • Approximation: Out_tok/s / In_tok/s ≈ OSL / ISL (exact ratio is U / (BS − U), converges for large values)
06 — Etched's Assumptions

What Etched assumes in their analysis

  • ISL=2048, OSL=128 — described as "NVIDIA and AMD's standard benchmark." Long input, short output. Minimizes decode users and KV cache traffic. Maximizes arithmetic intensity.
  • Batch size 2175 — ISL + (OSL−1) = 2048 + 127. With ISL=2048 this means 1 prefill user + 127 decode users per iteration.
  • Infinite on-chip compute — their methodology assumes HBM bandwidth is the only constraint, then calculates the TFLOPS needed to saturate it. This yields a large number (6.8 PFLOPS), which they present as evidence that inference is "compute-bound."
  • FP8 model weights, dense — 68.45 GB claimed, but their math uses 70B × 2 bytes = 140 GB, which is FP16 storage. "FP8 precision" refers to the compute datatype, not HBM storage format.
  • FP8 KV cache — claimed, but their KV formula uses × 2 × 2 (last ×2 = 2 bytes per value = FP16). Actual HBM reads are at FP16.
  • 64 transformer layers — Etched's KV formula uses 64 layers. Llama 3 70B has 80 layers. This underestimates KV cache by 1.25×.
  • Perfect parallelization across 8 chips — no communication overhead, no all-reduce latency, no load imbalance.
  • HBM3 at 3.35 TB/s per chip — total system bandwidth 26.8 TB/s across 8 chips.
  • Sohu peak TFLOPS: undisclosed — they claim ">90% FLOPS utilization" but never publish the denominator.
07 — Reproducing Etched's Exact Formula

Etched's verbatim arithmetic from the blog post

  • FLOPs: (2048 + 127) × 70B × 2 = 304 TFLOPS — correct
  • Weights: 70B × 2 bytes = 140 GB — uses 2 bytes/param (FP16), not FP8
  • KV cache: 127 × 64 × 8 × 128 × 2175 × 2 × 2 = 72 GB — uses 64 layers and 2 bytes/val (FP16)
  • Arithmetic intensity: 304T / (140 + 72) GB = 1,434 FLOPs/byte
  • H200 compute needed: 1,434 × 3.35 TB/s = 4.8 PFLOPS

Errors identified

  • Wrong layer count: KV formula uses n_layers = 64. Llama 3 70B has 80 layers. Underestimates KV cache by 1.25× (72 GB → 90.5 GB corrected).
  • Precision mismatch: Blog header says "FP8 precision" but every formula uses × 2 bytes — that's FP16 storage. FP8 is the compute datatype for matmuls; HBM reads are at FP16.
  • Uniform context: Uses (2048 + 127) = 2175 for all 127 decode users. In staggered continuous batching, user i has context (2048 + i). Minor effect: overestimates KV by ~1.5% at this config.
